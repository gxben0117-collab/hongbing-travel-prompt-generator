# 藍寶出圖流程紀錄

日期：2026-05-19

## 今日做過的事

今天主要在測試「藍寶原圖」透過「紅兵旅拍大片生成器」產生旅拍大片咒語，再送到 OpenAI 圖像 API 出圖，最後把圖片放回「藍寶輸出圖」。

來源圖片資料夾：

```text
C:\Users\User\Documents\Codex\藍寶原圖
```

輸出圖片資料夾：

```text
C:\Users\User\Documents\Codex\藍寶輸出圖
```

使用的生成器：

```text
C:\Users\User\Documents\Codex\紅兵旅拍大片生成器
```

## 曾經走偏的地方

今天一開始我額外加入了很多自己的規則，例如：

- A4 蝴蝶書比例控制
- 人物置中
- 安全邊
- 人物全身
- 臉在畫面上三分之一
- 額外安全服裝限制
- 額外構圖規則
- v2 / v3 隨機構圖嘗試

這些做法雖然是想解決印刷比例、頭身比例和安全審核問題，但實際上蓋掉了「紅兵旅拍大片生成器」原本已經寫好的咒語控制，導致成品變得太平、太固定、人物常佔畫面 2/3，失去原本程式的旅拍大片感。

這不是正確方向。

## 錯誤原因備註

前面做錯的核心原因是：沒有完全尊重「紅兵旅拍大片生成器」原本的咒語控制，而是繞過了程式正式產咒語的流程。

具體錯誤如下：

1. 沒有直接使用程式內的 `buildEnglishPrompt()` 當作唯一咒語來源。

   早期批次腳本只讀取 `src/data.js` 的風格資料，再由我另外拼接咒語。這等於只拿到風格素材，卻沒有使用使用者原本在 `src/app.js` 裡寫好的完整咒語控制、身份鎖、動作控制、妝容控制、品質控制和負面詞控制。

2. 額外加入 A4、構圖、安全邊等限制，蓋掉原本的隨機效果。

   我加入了人物置中、全身、臉在上三分之一、安全邊、固定 A4 蝴蝶書比例等指令。這些指令權重太強，導致模型把不同風格都壓成相似構圖，人物常固定佔畫面 2/3，看起來平凡、模板化。

3. 失敗後擅自修改咒語再重送。

   有些任務被安全系統擋下後，我曾經補上更保守的服裝和姿態限制再送。這違反使用者後來確認的規則：失敗就只記錄，不改咒語、不重送、不補跑。

4. 把「避免風險」誤處理成「降低大片感」。

   使用者要的是旅拍大片、漂亮、華麗、有場景、有電影感，隨機風格裡也包含科幻、賽博龐克、星空、仙俠、奇幻等系列。我額外加的安全和構圖限制讓畫面變保守、普通，削弱了原程式本來的大片感。

5. 為了修印刷比例問題，誤把流程改成自己的新流程。

   A4 蝴蝶書比例可以作為後製或另外專案規格討論，但不能取代這次的核心流程。這次核心需求是：用原圖，讓紅兵旅拍大片生成器自己隨機產咒語，再送 API 出圖。

後續避免再犯的原則：

```text
不要自行拼咒語。
不要自行加構圖限制。
不要自行加安全補詞。
不要失敗後改咒語重送。
只使用程式原生咒語流程。
```

## 使用者確認的正確方向

使用者原本做出來的效果不錯，因為程式內部本來就已經寫好咒語控制。

詳細正式流程文件：

```text
C:\Users\User\Documents\Codex\紅兵旅拍大片生成器\docs\project\藍寶Native出圖正式流程.md
```

後續正確方向是：

```text
原圖 → 紅兵旅拍大片生成器原生隨機咒語 → OpenAI 圖像 API → 成功圖放到藍寶輸出圖
```

重點：

- 不要額外改咒語。
- 不要額外補安全詞。
- 不要額外補 A4 構圖規則。
- 不要改成自己的構圖邏輯。
- 不要為了失敗任務重寫咒語再送。
- 不要為了湊數另外抽圖補跑。

## 正確批次規則

每一張來源圖只嘗試一次：

```text
原圖 x1 → 程式原生隨機咒語 x1 → API 出圖 x1
```

成功：

```text
把圖片存到 C:\Users\User\Documents\Codex\藍寶輸出圖
```

失敗：

```text
只記錄到失敗清單，不修改咒語，不重送。
```
